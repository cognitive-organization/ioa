### DEC-2026-05-14-008

- **Date:** 2026-05-14
- **Project:** FINANFIX (paribali-labs/finanfix)
- **Decision:** Adopt "zero-progress alarm" as the canonical observability pattern for any cron that loops over candidates: assert `if (workDone === 0 && candidates >= threshold) → Slack critical + Sentry with tag alert:zero-progress`. First applied to cron `categorize-transactions` (threshold 5), then rolled out to `send-notifications` (threshold 5 for in_app, 20 for queue) and `proactive-alerts` (critical when both RPCs fail, warning when one).
- **Context:** A 7-day silent outage (2026-05-07 → 2026-05-14) of `categorize-transactions` returning 200 OK with zero work done and only a `logWarn` exposed the gap. The pattern asserts on absence of work, not on errors, catching the entire class of bugs where the route runs but doesn't progress.
- **Alternatives:** Rely on Sentry breadcrumbs and dashboards; build a generic cron heartbeat service; require every cron to publish a "rows touched" metric instead of an alarm
