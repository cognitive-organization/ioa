### DEC-2026-05-13-001

- **Date:** 2026-05-13
- **Project:** FINANFIX (paribali-labs/finanfix)
- **Decision:** Remove the `AND a.account_type <> 'credit_card'` filter from the `get_period_flow_summary` and `get_monthly_flow_aggregates` RPCs (migration 056), so credit-card purchases count as outflow in the dashboard summary like they already did in the Activity screen and the TS fallback.

- **Alternatives:** Patch the TS fallback to match the buggy RPC behavior; route the dashboard to read from the Activity screen aggregator

### DEC-2026-05-13-002

- **Date:** 2026-05-13
- **Project:** FINANFIX (paribali-labs/finanfix)
- **Decision:** Make `isCreditCardBillPayment` symmetric — detect both sides of the operation (credit on the card account and debit on the checking/savings account with a specific invoice-payment description) rather than relying solely on the card side and the Pluggy category.
- **Context:** The detection only caught the card side; the checking-account counterpart depended on the provider's category and could be recorded as a regular expense, inflating monthly totals for any user whose bank described the transaction differently.
- **Alternatives:** Trust the Pluggy category exclusively; ship a separate categorization rule per bank provider
- **Made by:** Human (Luan Coudray) + Agent (Claude Opus 4.7)
- **Source:** PR #435 / commit 83cdc31
- **Review date:** 2026-06-12

### DEC-2026-05-13-003

- **Date:** 2026-05-13
- **Project:** FINANFIX (paribali-labs/finanfix)
- **Decision:** Enforce the invariant "amount is always stored positive in the DB" by applying `Math.abs` across the four sync paths (webhook, sync route, callback, admin import) and normalizing the dedup `contentKey` accordingly.
- **Context:** Reversals and adjustments from the provider could arrive with inverted signs and corrupt downstream sums; the transaction `type` (debit/credit) is now the single source of truth for direction, leaving `amount` to carry magnitude only.
- **Alternatives:** Add a per-source normalization layer only at read time; reject negative-amount inserts at the DB level
- **Made by:** Human (Luan Coudray) + Agent (Claude Opus 4.7)
- **Source:** PR #435 / commit 83cdc31
- **Review date:** 2026-06-12
